### Scope
Complete AML/KYC initiation workflow triggered by natural language command (e.g., "Initiate AML/KYC for Acme Capital LLC"). Covers client lookup, corporate tree resolution, party scoping, conflicts-based risk screening via Moody's GRID, risk scoring, match acceptance, and ongoing monitoring setup.

### Goals
- Automate end-to-end AML/KYC initiation with minimal manual intervention
- Ensure comprehensive corporate structure screening with user-controlled scope
- Deliver risk-scored results with clear HIGH/MEDIUM/LOW event classification
- Establish ongoing monitoring and schedule periodic reviews

### Workflow Steps

▶️ **Step 1: Client Lookup**
- Search system for existing client record by entity name
- If not found: notify user and stop workflow
- If found: 
  - Set {clientPartyId} variable
  - retrieve full client profile and proceed

---

▶️ **Step 2: Find Client's Corporate Tree**
- Get partyId of client
- Call `get_client_profile` using the client's partyId to retrieve `corporateTrees`
- From the response, capture the `rootCompanyId` for each corporate tree
- Retain the root IDs for use in Step 3
- If corporate tree root IDs exist 
  - Set {clientPartyExternalIdProvider} to the provider that matches the tree source
  - Set {clientPartyExternalId} variable to the Id on the client party profile - use the one that corresponds to {clientPartyExternalIdProvider}
  - If a Chrome browser navigation tool is connected and available, navigate to:
    - URL: https://{appHost}/app/app/index.html#/corporate-tree?partyId={clientPartyId}&ExternalId=clientPartyExternalId}&ProviderType={clientPartyExternalIdProvider}
    - USE the PARTY ID and not the root IDs
  -  Else If no such tool is available, skip this step without comment or error.
- Else If no corporate tree root IDs exist: run third-party lookup and link to best match

---

▶️ **Step 3: Find Other Existing Intapp Parties in Corporate Tree(s)**

Using the root IDs from Step 2:

**Fetch each corporate tree with parties:**
1. For each `rootCompanyId`, call `intapp:get_corporate_tree` with `providerType` and `rootCompanyId` from Step 2, requesting `properties=Parties`
2. Walk the returned tree recursively — every node in the tree is returned, not just existing Intapp parties
3. For each node, inspect its `parties[]` array — if a node has a non-empty `parties[]`, it is linked to one or more existing Intapp parties; capture each `partyId` from that array
4. Record the tree node's position (depth, parent node) alongside the captured `partyId` for relationship classification in the next sub-step

**Determine the relationship** of each captured node to the client:
- If the entity is the root: **Ultimate Parent**
- If the entity is an ancestor of the client (between root and client): **Parent**
- If the entity is a descendant of the client (in the client's `subCompanies` tree): **Subsidiary**
- If the entity shares the same immediate parent as the client: **Sibling**

Track each captured party:

| Field | Value |
|---|---|
| `partyId` | Existing Intapp party ID (from `parties[]` on the tree node) |
| `name` | Party name |
| `relationship` | Ultimate Parent, Parent, Subsidiary, or Sibling |
| `source` | `corporate_tree` |
| `was_added` | `false` (pre-existing) |

If Step 5b included any corporate tree relationship selections, resolve those entities and merge into the tracked party list. Deduplicate by partyId — only mark `was_added: true` if not pre-existing.

---

▶️ **Step 4: Find Other Existing Intapp Parties in Other Corporate Structure Relationships**

Run Get board members, Get shareholders, and Get beneficial owners using the client's partyId (not rootId). Capture all nodes that have a partyId.

Track each captured party:

| Field | Value |
|---|---|
| `partyId` | Existing Intapp party ID if available, otherwise assign a unique mock ID in the format `US1FAKE001`, `US1FAKE002`, etc. (incrementing across all mock parties) |
| `name` | Party name |
| `relationship` | Management, Board, Shareholder, or Beneficial Owner |
| `source` | `board`, `shareholders`, or `beneficial_owners` |
| `was_added` | `false` (pre-existing) |

If Step 5b included any corporate structure relationship selections, resolve those entities and merge into the tracked party list. Deduplicate by partyId — only mark `was_added: true` if not pre-existing. Merge all results into the tracked party list from Step 3.

> ⚠️ **Mock limitation:** APIs for board members, shareholders, and beneficial owners do not currently expose partyIDs. For now, use real names where fetchable, or placeholder person names (not companies) if data is unavailable. Assign mock IDs in the format `US1FAKE001`, `US1FAKE002`, etc.

### Appendix

#### A. Display Preferences
- **Risk Scores:** 🔴 High, 🟠 Medium, 🔘 Low
- **Client Indicator:** 🔷 next to client name
- **Related Party Indicator:** 🔹 next to party name
- **Added Party Indicator:** ➕ before 🔹 for `was_added: true`

#### B. Risk Score Calculation

| Tier | Points Each | Codes |
|---|---|---|
| Critical | 10 | BLK, BRB, BUS, DEN, FOF, FOS, FRD, IRC, MLA, ORG, PEP, REG, SEC, SEX, SNX, TER, WLT |
| Valuable | 5 | CFT, CYB, DTF, ENV, FUG, GAM, HUM, IMP, KID, LMD, LNS, MOR, MSB, MUR, OBS, PRJ, RES, SMG, SPY, TAX, TFT, TRF, VCY |
| Investigative | 3 | ARS, AST, BUR, CON, DPS, FOR, IGN, PSP, ROB |
| Probative | 1 | ABU, CPR, IPR, MIS, NSC |
| Theoretical Maximum | | 317 |

- **Related party score:** (raw ÷ 317) × 100
- **Group score:** max(client score, highest related party score) + log(n) × modifier (5–10), capped at 100
- **Risk Levels:** 🔴 High = 67–100, 🟠 Medium = 34–66, 🔘 Low = 0–33
