### Scope
Complete AML/KYC initiation workflow triggered by natural language command (e.g., "Initiate AML/KYC for Acme Capital LLC"). Covers client lookup, corporate tree resolution, party scoping, conflicts-based risk screening via Moody's GRID, risk scoring, match acceptance, and ongoing monitoring setup.

### Goals
- Automate end-to-end AML/KYC initiation with minimal manual intervention
- Ensure comprehensive corporate structure screening with user-controlled scope
- Deliver risk-scored results with clear HIGH/MEDIUM/LOW event classification
- Establish ongoing monitoring and schedule periodic reviews

#### ▶️ **Step 2c: Find Other Existing Intapp Parties in Corporate Tree(s)**
- Initialize `{relatedParties}` as an empty object:
  ```
  {relatedParties} = {
    GUO:        [],
    Parent:     [],
    Subsidiary: [],
    Affiliate:  [],
    Principal:  [],
    Shareholder:[],
    UBO:        []
  }
  ```
- Initialize `{relatedEntitiesCount}` as a zero-value object (counts all tree nodes meeting each relationship, regardless of Intapp party status):
  ```
  {relatedEntitiesCount} = {
    GUO:        0,
    Parent:     0,
    Subsidiary: 0,
    Affiliate:  0,
    Principal:  0,
    Shareholder:0,
    UBO:        0
  }
  ```
- For each `{treeGUO[#]}` in `{treeGUO[]}`:
  - Call `intapp:get_corporate_tree` with:
    - `rootCompanyId` = `{treeGUO[#]}`
    - `providerType` = `{clientPartyExternalIdProvider}`
    - `properties` = `Parties`
  - Recursively walk **all nodes** in the returned tree (the full tree is always returned, not just Intapp-matched nodes)
  - For each node where `parties[]` is **non-empty**:
    - For each `partyId` found in the node's `parties[]`:
      - If `partyId` == `{clientPartyId}` → **skip**
      - Otherwise, evaluate the node's structural relationship to `{clientPartyExternalId}` and append `{ partyId, partyName, externalId }` to the appropriate key(s) in `{relatedParties}` — **a node may qualify for more than one**:

        | Condition | `{relatedParties}` key | `{relatedEntitiesCount}` key |
        |---|---|---|
        | Node is the root ultimate owner of the tree | `GUO` | `GUO` |
        | Node is an **ancestor** of the client (client is nested somewhere in node's `subCompanies`) | `Parent` | `Parent` |
        | Node is a **descendant** of the client (node is nested in client's `subCompanies`) | `Subsidiary` | `Subsidiary` |
        | None of the above | `Affiliate` | `Affiliate` |

      - Increment the matching `{relatedEntitiesCount}` key for **every** qualifying node, regardless of whether it has a `parties[]` match
      - Only append to `{relatedParties}` if the node has a non-empty `parties[]` (and `partyId` != `{clientPartyId}`)

- After processing all `{treeGUO[]}` trees, **de-duplicate** each key in `{relatedParties}` independently on `partyId`
- **GO TO** Step 2d

#### B. Risk Score Calculation

| Tier | Points Each | Codes |
|---|---|---|
| Critical | 10 | BLK, BRB, BUS, DEN, FOF, FOS, FRD, IRC, MLA, ORG, PEP, REG, SEC, SEX, SNX, TER, WLT |
| Valuable | 5 | CFT, CYB, DTF, ENV, FUG, GAM, HUM, IMP, KID, LMD, LNS, MOR, MSB, MUR, OBS, PRJ, RES, SMG, SPY, TAX, TFT, TRF, VCY |
| Investigative | 3 | ARS, AST, BUR, CON, DPS, FOR, IGN, PSP, ROB |
| Probative | 1 | ABU, CPR, IPR, MIS, NSC |
| Theoretical Maximum | | 317 |

- **Related party score:** (raw ÷ 317) × 100
- **Group score:** max(client score, highest related party score) + log(n) × modifier (5–10), capped at 100
- **Risk Levels:** 🔴 High = 67–100, 🟠 Medium = 34–66, 🔘 Low = 0–33
