### Scope
Complete AML/KYC initiation workflow triggered by natural language command (e.g., "Initiate AML/KYC for Acme Capital LLC"). Covers client lookup, corporate tree resolution, party scoping, conflicts-based risk screening via Moody's GRID, risk scoring, match acceptance, and ongoing monitoring setup.

### Goals
- Automate end-to-end AML/KYC initiation with minimal manual intervention
- Ensure comprehensive corporate structure screening with user-controlled scope
- Deliver risk-scored results with clear HIGH/MEDIUM/LOW event classification
- Establish ongoing monitoring and schedule periodic reviews

- For each `{treeGUO[#]}` in `{treeGUO[]}`:
  - Call `intapp:get_corporate_tree` with:
    - `rootCompanyId` = `{treeGUO[#]}`
    - `providerType` = `{clientPartyExternalIdProvider}`

- **Pass 1 — Flat scan for party-bearing nodes:** Scan the entire tree JSON response in a single pass. Collect every node where `parties[]` is non-empty into a candidate list `{ externalId, name, partyId }`. Exclude the client node (`externalId` == `{clientPartyExternalId}`).
- **Pass 2 — Classify each candidate using parent-chain walks:**
    - First, build a parent index by traversing the tree top-down through `subCompanies[]`. For every node record its parent's `externalId`.
    - Then evaluate all four conditions below independently for each candidate. Collect every matching bucket label into a set — do not stop at the first match:
      - **GUO:** candidate `externalId` == `{treeGUO[#]}`
      - **Parent:** Starting from the client node, walk up via the parent index. If this candidate appears anywhere on that path, label Parent.
      - **Subsidiary:** Starting from the candidate node, walk up via the parent index. If the client node is found exactly 1 step away, label Subsidiary. If found >1 step away, label Subsidiary. Both are the same bucket.
      - **Sibling:** None of GUO, Parent, or Subsidiary matched.
    - After all labels are determined for the candidate, increment `total` and (if `parties[]` non-empty) `parties` and `list` once per matching bucket — one increment per bucket, applied after classification is complete.
  - Evaluate all four conditions independently for every candidate. A candidate may match multiple — increment every matching bucket. Do not stop at the first match.

#### B. Risk Score Calculation

| Tier | Points Each | Codes |
|---|---|---|
| Critical | 10 | BLK, BRB, BUS, DEN, FOF, FOS, FRD, IRC, MLA, ORG, PEP, REG, SEC, SEX, SNX, TER, WLT |
| Valuable | 5 | CFT, CYB, DTF, ENV, FUG, GAM, HUM, IMP, KID, LMD, LNS, MOR, MSB, MUR, OBS, PRJ, RES, SMG, SPY, TAX, TFT, TRF, VCY |
| Investigative | 3 | ARS, AST, BUR, CON, DPS, FOR, IGN, PSP, ROB |
| Probative | 1 | ABU, CPR, IPR, MIS, NSC |
| Theoretical Maximum | | 317 |

- **Related party score:** (raw ÷ 317) × 100
- **Group score:** max(client score, highest related party score) + log(n) × modifier (5–10), capped at 100
- **Risk Levels:** 🔴 High = 67–100, 🟠 Medium = 34–66, 🔘 Low = 0–33

### Appendix E. Multi-Source Relationship Rule

Relationship buckets are **non-exclusive** — a single entity can and should be recorded
in multiple buckets simultaneously. Evaluate all bucket conditions for every entity
regardless of source; do not stop at the first match.

**Example:** The GUO is always also a Parent. If it additionally appears in the
shareholders list, it holds three relationships at once: `["GUO", "Parent", "Shareholder"]`.
It appears as a single row in `{relatedPartiesTable[]}` with all three labels — never
as duplicate rows, and never counted more than once toward any bucket's `total`.
